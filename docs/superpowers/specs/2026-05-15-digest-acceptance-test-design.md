# Digest Acceptance Test Design

## 背景

`follow-news` 当前已经有数据采集、合并、摘要输入展示、PDF/HTML 转换等测试，但最终日报正文主要由 `references/digest-prompt.md` 和 agent 按模板生成。这个环节缺少一个稳定的产品验收测试：当 prompt、template、fixture 或输出脚本变化时，项目无法自动发现最终 Markdown/Discord 日报格式被无意改坏。

本设计的目标是在不改变现有日报输出格式的前提下，把当前产品输出固化成可回归的 acceptance contract。第一版只覆盖 Markdown/Discord 输出，不覆盖 Email HTML、PDF 渲染、网络采集、LLM 真实生成质量或发送渠道。

## 目标

- 提供默认 CI 可运行的确定性验收测试，验证最终 Markdown/Discord 日报格式。
- 使用固定 merged JSON fixture 和固定日期，避免网络、模型、当天日期和时区导致测试抖动。
- 使用 golden file 作为产品验收样例，格式变化可以通过 review 明确看见。
- 支持显式 golden 更新流程：只有设置 `UPDATE_GOLDEN=1` 时才覆盖 golden。
- 提供真实 Codex 手动验收入口，用同一份 fixture 生成实际日报并和 expected 输出对比。

## 非目标

- 不重新设计日报文案、章节结构、排序规则或视觉风格。
- 不把真实 Codex/LLM 调用放进默认 CI。
- 不测试真实网络 pipeline、source fetcher、deduplication 或 scoring 算法。
- 不覆盖 Email HTML 和 PDF 的视觉/渲染验收。
- 不把 acceptance renderer 作为生产日报生成器。

## 推荐方案

采用双层验收：

1. 默认层：本地确定性 Markdown/Discord golden test。
2. 手动层：生成 Codex 验收上下文，让 Codex 按真实 prompt 生成日报，再由人工或 diff 检查。

默认层适合 CI，手动层适合发布前产品检查。这样可以同时获得稳定回归测试和真实 agent 行为反馈，但不会让不可控的 LLM 输出影响常规测试。

## 文件设计

新增文件：

- `scripts/render-acceptance-digest.py`
  - 只服务 acceptance test。
  - 读取 merged JSON fixture。
  - 读取 `config/defaults/topics.json` 获取 topic emoji 和 label。
  - 按当前 `references/digest-prompt.md` 与 `references/templates/discord.md` 的格式 contract 渲染 Markdown/Discord。
  - 支持参数：`--input`、`--topics`、`--date`、`--version`、`--output`。
  - 不调用网络，不调用 LLM，不写 archive。

- `tests/test_acceptance_digest.py`
  - 调用 renderer 的函数或 CLI。
  - 将生成结果与 golden 文件逐字节比较。
  - 失败时输出 unified diff。
  - 当 `UPDATE_GOLDEN=1` 时更新 golden 并通过测试。
  - 额外保留少量结构断言，避免 golden 被错误更新后仍然掩盖明显结构问题。

- `tests/golden/daily-discord.md`
  - 固定的期望日报正文。
  - 用于 CI diff，也作为产品验收样例。

- `tests/fixtures/acceptance-merged.json`
  - 推荐新增一个专用 fixture，而不是复用 `tests/fixtures/merged.json`。
  - 专用 fixture 应更小、更稳定，并覆盖最终输出边界：
    - topic articles
    - multi-source indicator
    - Twitter/KOL metrics
    - GitHub releases
    - GitHub Trending
    - Blog Picks
    - Podcast Remix
    - footer source counts

## 命令设计

默认验收：

```bash
python -m unittest tests.test_acceptance_digest -v
```

显式更新 golden：

```bash
UPDATE_GOLDEN=1 python -m unittest tests.test_acceptance_digest -v
```

手动生成验收样例：

```bash
python scripts/render-acceptance-digest.py \
  --input tests/fixtures/acceptance-merged.json \
  --topics config/defaults/topics.json \
  --date 2026-02-27 \
  --version 3.17.0 \
  --output /tmp/follow-news-acceptance/expected.md
```

## 格式 Contract

第一版 contract 只固化当前输出，不引入新格式。

标题沿用当前 Discord 模板：

```markdown
# 🚀 Tech Digest - {{DATE}}
```

Topic section 沿用当前结构：

```markdown
## {{emoji}} {{label}}

• 🔥{{quality_score}} | {{title}}
  <{{link}}>
  *[{{source_count}} sources]*
```

topic 选择与排序规则：

- section 顺序按 `topics.json`。
- topic 内按 `quality_score` 降序。
- 只输出 `quality_score >= 5` 的文章。
- 不做额外重排、分组或标题改写。
- 每个 article item 必须有 source link。
- `multi_source` 为 true 时显示 source count。

fixed sections 按当前 prompt 规则渲染；第一版只在 fixture 存在对应数据时输出：

- `📢 KOL Updates`
- `📦 GitHub Releases`
- `🐙 GitHub Trending`
- `📝 Blog Picks`
- `🎙️ Podcast Remix`

footer 沿用当前模板：

```markdown
---
📊 Data Sources: RSS {{rss}} | Twitter {{twitter}} | Reddit {{reddit}} | Web {{web}} | GitHub {{github}} releases + {{trending}} trending | Podcast {{podcast}} episodes | Dedup: {{merged}} articles
🤖 Generated by follow-news v{{version}} | <https://github.com/tangwz/follow-news> | Powered by OpenClaw
```

## 断言策略

主断言：

- 生成 Markdown 与 `tests/golden/daily-discord.md` 完全一致。
- 失败时输出 unified diff，便于 code review 判断是否是预期格式变化。

结构断言：

- 标题存在。
- 至少一个 topic section 存在。
- topic article item 同时包含 `🔥` 和 `<http...>`。
- topic section 内分数非递增。
- footer 存在。
- fixed section 如果 fixture 中有对应数据，则输出中必须出现对应 header。

golden 更新策略：

- 默认测试绝不写 golden。
- 只有 `UPDATE_GOLDEN=1` 时覆盖 `tests/golden/daily-discord.md`。
- 更新 golden 时测试输出应明确提示 golden 已更新。

## 手动 Codex 验收入口

手动验收不进入默认 CI。第一版提供一个生成验收上下文的流程，输出到 `/tmp/follow-news-acceptance/`：

- `merged.json`
- `summarized.txt`
- `prompt.md`
- `expected.md`

`prompt.md` 应明确：

- 不运行网络 pipeline。
- 使用提供的 `merged.json`。
- 使用当前 `references/digest-prompt.md` 的 Markdown/Discord 输出规则。
- 报告日期固定。
- 输出保存为 `actual.md`。

人工或后续脚本比较：

```bash
diff -u /tmp/follow-news-acceptance/expected.md /tmp/follow-news-acceptance/actual.md
```

这个流程验证真实 Codex 行为，但不要求 Codex 输出逐字等于 deterministic renderer。产品验收时重点检查章节顺序、条目格式、链接、分数、fixed sections 和 footer 是否符合 contract。

## 风险与缓解

- 风险：renderer 变成第二套生产实现。
  - 缓解：明确 renderer 只服务 acceptance test，不进入 digest prompt 或交付路径。

- 风险：golden 文件被机械更新，掩盖格式回归。
  - 缓解：保留结构断言，并要求 golden diff 进入 code review。

- 风险：真实 Codex 手动验收和确定性 renderer 输出不完全一致。
  - 缓解：默认 CI 只验 contract；真实 Codex 验收作为发布前人工检查，不要求逐字一致。

- 风险：fixture 覆盖不足。
  - 缓解：使用专用 `acceptance-merged.json`，显式覆盖 topic、multi-source、KOL、GitHub、Trending、Blog Picks、Podcast 和 footer counts。

## 实施顺序

1. 新增专用 acceptance fixture。
2. 新增 deterministic renderer。
3. 新增 golden 文件。
4. 新增 unittest，包括 golden diff 和结构断言。
5. 文档化默认验收和 `UPDATE_GOLDEN=1` 更新命令。
6. 新增或文档化 Codex 手动验收上下文生成流程。

## 验证方式

实现后应运行：

```bash
python -m unittest tests.test_acceptance_digest -v
python -m unittest discover -s tests -v
```

有意变更格式时应运行：

```bash
UPDATE_GOLDEN=1 python -m unittest tests.test_acceptance_digest -v
git diff -- tests/golden/daily-discord.md
```
